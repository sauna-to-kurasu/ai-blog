## 前提：僕のClaude Code歴

簡単に自己紹介だけ。

## 7つの「プロンプトを変えた瞬間」

これから紹介する7つは、すべて実際にハマって、手戻りが発生して、「次からはこう伝えよう」と決めた瞬間です。

数字は厳密な計測値ではなく体感ベースですが、方向性は間違いないと思います。

---

### 瞬間1 —「いい感じにして」

**何が起きたか**

サウナブログを作り始めた頃、トップページのデザインをAIに「いい感じにして」と伝えました。

返ってきたのは、Tailwind CSSのデフォルトカラー（#2563ebの青色）がベタ塗りされた、自分のイメージとまったく違うサイト。「違うな…」と思ってやり直しを頼むも、今度は別のデフォルトカラーに。これを5〜6回繰り返して、半日が溶けました。

**気づき**

「いい感じ」はAIにとって情報量ゼロ。自分が何を「いい」と思っているのか、言語化できていなかった。カラーコードもフォントサイズも、全部「僕の頭の中にしかない情報」だった。

**改善後のプロンプト**

<AiChat messages={[
  { role: "human", text: "トップページのヒーローセクションを変更して。カラーは背景#0F172A、アクセント#22D3EE。Tailwind標準の青(#2563eb等)は使わないで。落ち着いたダーク系のトーンで。" },
  { role: "ai", text: "承知しました。#0F172Aを背景、#22D3EEをアクセントカラーとして、ヒーローセクションのデザインを変更します。" }
]} />

**体感データ**: デザイン修正回数が1セクションあたり5〜6回 → 1〜2回に。

---

### 瞬間2 —「このエラー、直して」

**何が起きたか**

本番デプロイ後、CSSが適用されない問題が発生しました。「デプロイしたらデザインが崩れた、直して」とだけ伝えました。

Claude Codeはローカルのコードを確認して「問題ありません」と返してくる。**当然です。** 本番環境で起きている問題を、AIはブラウザ越しに見ることができません。

結局、原因はpackage.jsonにTailwind CSSの依存関係が含まれていなかったこと。ローカルでは動いていたけど、Cloudflare Pagesのビルド環境ではインストールされていなかった。

**気づき**

AIはコードを読めるけど、画面は見えない。「どの環境で」「何が起きているか」を伝えないと、的外れな回答しか返ってこない。

**改善後のプロンプト**

<AiChat messages={[
  { role: "human", text: "本番環境(Cloudflare Pages)でTailwind CSSが適用されていない。ローカル(pnpm dev)では正常に表示される。ビルドログにエラーはなし。package.jsonにtailwindcssの依存関係が含まれているか確認してほしい。" },
  { role: "ai", text: "package.jsonを確認します。…tailwindcssがdevDependenciesに含まれていませんね。pnpm add -D tailwindcss を実行します。" }
]} />

**体感データ**: エラー解決のやりとりが6〜7往復 → 1〜2往復に。

### 瞬間3 —「全ページのデザインを統一して」

**何が起きたか**

プロフィールページのデザインが気に入ったので、「他の全ページもこのデザインにして」と指示。

でも「このデザイン」がAIには伝わらない。色のことなのか、レイアウトなのか、フォントサイズなのか。結果、中途半端な変更が5つ以上のファイルに散らばって、どこが変わったのか自分でも把握できなくなりました。

結局、9回の段階的な修正を経て、やっとデザインルールを整理できました。

**気づき**

「全部」「統一」という言葉は人間同士の会話では通じるけど、AIにとっては範囲が広すぎる。**何を基準にするか（参照ファイル）** と **何を変えるか（カラー、サイズ、レイアウト）** を分けて伝える。そして**1ファイルずつ確認しながら進める**のが最短距離だった。

**改善後のプロンプト**

<AiChat messages={[
  { role: "human", text: "BlogPost.astroの記事本文のスタイルを修正してほしい。基準はProfileArticle.jsxのデザイン。変更点は3つ。(1)本文の文字サイズを17px (2)カラーパレットを#0E7490ベース (3)カード型レイアウトをgrid-cols-1 md:grid-cols-2に。まずBlogPost.astroだけ変更して、確認してから他に広げたい。" },
  { role: "ai", text: "BlogPost.astroの3点を修正します。他のファイルへの展開は確認後に行いますね。" }
]} />

**体感データ**: 一度に5ファイル以上変更して手戻り → 1ファイルずつ確認で手戻りほぼゼロ。

---

### 瞬間4 —「長文で全部詰め込んだプロンプト」

**何が起きたか**

記事を書いてもらうためのプロンプトファイルを作りました。v1からv5まで、5回書き直しました。

問題は「足し算」の発想で作っていたこと。WordPress時代のSwell対応のHTMLコメント形式がそのまま残っていたり、「品質最優先」と書きつつ「スピード重視で」という矛盾する指示が同居していたり。

AIは矛盾する指示を受けると、どちらを優先すべきか分からず、結果が中途半端になります。

**気づき**

プロンプトは「足し算」ではなく「引き算」。環境が変わったら（WordPress → Astro、Cursor → Claude Code）、古い指示を消す。矛盾する指示を放置しない。

<div class="highlight-box">
  <strong>プロンプトの棚卸しルール（僕が実践しているもの）</strong>
  <ul>
    <li>環境が変わったら、プロンプト内の環境依存部分を全削除する</li>
    <li>「最優先」「絶対」が2つ以上あったら、本当の最優先を1つだけ残す</li>
    <li>1つのプロンプトに3つ以上の目的を詰め込まない</li>
  </ul>
</div>

**体感データ**: プロンプトファイル書き直し5回。原因の半分は「古い指示の残骸」。

### 瞬間5 —「記事を書いて」

**何が起きたか**

サウナブログの記事を「ホームサウナの導入記事を書いて」とだけ伝えました。

返ってきたのは、ネットで拾える一般的なサウナ情報を並べた薄い記事。当然です。**僕の家のサウナの電気代が月いくらか、導入に何ヶ月かかったか、断熱等級がいくつかなんて、AIは知りようがない。**

一次情報を渡さなければ、誰でも書けるありきたりな記事しか出てこない。

**気づき**

AIは世の中の一般知識は大量に持っている。でも「僕だけが知っている情報」はゼロ。一次情報（自分の体験データ、コスト、期間、感想）を渡さなければ、そのブログを読む理由がない記事になる。

もう一つ大きかったのは、**「情報が足りなければ質問してくれ」とルールに加えた**こと。これだけで、AIが勝手に薄い記事を書き始めるのを防げました。

**改善後のプロンプト**

<AiChat messages={[
  { role: "human", text: "docs/PROJECT_CONTEXT.mdを読んで、記事No.5を書いて。僕のサウナの電気代データ（月額約3,000円）、導入期間（問い合わせから7ヶ月）、断熱等級7の家の仕様を必ず含めて。情報が足りなければ質問して。" },
  { role: "ai", text: "PROJECT_CONTEXT.mdを確認しました。追加で確認させてください。(1)サウナの日常的な使用頻度は？ (2)太陽光発電との相殺後の実質電気代は？…" }
]} />

**体感データ**: AIからの質問数 — コンテキストなし: 0問（薄い記事をそのまま書く） → コンテキスト+質問ルールあり: 5〜8問（深い記事になる）。

### 瞬間6 —「スマホでも見やすくして」

**何が起きたか**

レスポンシブ対応を「スマホでも見やすくして」と頼みました。AIは375px幅で調整してくれたけど、タブレット（768px）では崩れていた。さらに、修正後にAIが「完了しました」とcommitしようとした。**まだ自分で確認していないのに。**

**気づき**

「見やすくして」だけでは、**何を基準に「完了」とするか**がAIに伝わっていない。確認すべき画面サイズと、「人間が確認するまで完了じゃない」というルールを明示する必要があった。

これがきっかけで、CLAUDE.mdに「レスポンシブ確認は375px / 768px / 1280pxの3サイズ」「確認前にgit操作はしない」というルールを書き込みました。

**改善後のプロンプト**

<AiChat messages={[
  { role: "human", text: "レスポンシブを修正して。確認するのは375px、768px、1280pxの3サイズ。修正したらpnpm devで起動して、僕がブラウザで確認するまでgit操作はしないで。" },
  { role: "ai", text: "承知しました。3サイズで確認できるよう修正します。完了後にpnpm devを起動しますので、ブラウザで確認をお願いします。" }
]} />

**体感データ**: レスポンシブ修正のやり直し 3〜4回 → 1回で完了。

### 瞬間7 —「前と同じようにやって」

**何が起きたか**

セッションを跨いで作業を再開したとき、「前回の続きで、同じデザインルールでやって」と伝えました。

Claude Codeは**毎回まっさらな状態**で起動します。前回決めたカラーパレットも、フォントサイズも、「npmじゃなくてpnpmを使う」というルールも、何も覚えていません。

結果、Tailwind標準の青色が復活し、npmでパッケージがインストールされてlockファイルが二重になり、フォントサイズもバラバラに。前回やった作業の半分が巻き戻りました。

**気づき**

この瞬間、プロンプトの改善だけでは限界があることに気づきました。「前回伝えたこと」を毎回口頭で繰り返すのは無理。**ファイルに書いて、自動で読み込ませるしかない。**

振り返ると、7つの失敗には共通するパターンがありました。

      <p class="process-timeline-desc">エラーメッセージ、環境情報、ブラウザの表示状態、自分の一次データ。AIはコードを読めるけど、画面は見えない。「何が起きているか」を省略しない。</p>

  <div class="process-timeline-item">

## AIに聞いても出てこなかった「プロンプトの勘所」

最後に、この記事の核心を書きます。

### 自分のプロジェクト固有のNGリストは、AIに生成できない

「Tailwind標準の青は使うな」「npmは使うな、pnpmを使え」「確認前にcommitするな」——これらは全部、僕が実際にミスしてから決めたルールです。

AIに「プロンプトのコツを教えて」と聞けば、一般的なアドバイスは出てきます。でも、**自分のプロジェクトで何がNGなのか**は、自分で失敗しないと分からない。

### プロンプト改善は「足し算」ではなく「引き算」

AIに「プロンプトを改善するには？」と聞くと、「もっと詳しく書きましょう」「コンテキストを追加しましょう」と返ってきます。

実際には、**古い指示を削除する**ほうが効果が高かった。v1→v5の変遷で学んだのは「足すこと」じゃなく「矛盾を消すこと」でした。

### 一番効くプロンプト改善は「プロンプトを書かなくて済む仕組み」を作ること

毎回のプロンプトで「pnpmを使って」「この色は使わないで」と書くのは現実的じゃない。CLAUDE.mdに書いておけば、毎回のプロンプトはシンプルになる。

**プロンプトのコツを磨くより、仕組みを整えるほうが圧倒的に効果が高い。** これがこの記事で一番伝えたいことです。
