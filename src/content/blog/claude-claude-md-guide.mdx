## 結論：CLAUDE.mdは、AIへの「取扱説明書」

Claude Codeを使っていて、こんな経験はありませんか？

「さっきも言ったのに、また同じミスをする」「毎回ゼロから説明し直すのが面倒」

僕もそうでした。Claude Codeは優秀ですが、**何も伝えなければ毎回まっさらな状態で仕事を始めます。** 前回の会話で伝えたルールも、デザインの好みも、使っている技術も覚えていません。

CLAUDE.mdは、この問題を解決するファイルです。

## CLAUDE.mdとは何か

CLAUDE.mdは、**プロジェクトのルートフォルダに置くテキストファイル**です。Claude Codeは会話を始めるたびにこのファイルを自動で読み込んで、書かれたルールに従って動きます。

イメージとしては、新しいアルバイトに渡す「業務マニュアル」のようなものです。

- 「うちの店では、お客さんには敬語で話してね」 → **文体のルール**
- 「冷蔵庫の食材は左から使ってね」 → **ファイル管理のルール**
- 「レジ締めの手順はこの紙を見てね」 → **ワークフローの指示**

CLAUDE.mdがない状態は、マニュアルなしでアルバイトに仕事を任せるようなもの。毎回「あ、それはこうして」「いや、そうじゃなくて」と口頭で指示し直す必要があります。

僕は最初、CLAUDE.mdなしでClaude Codeを使っていました。そのときの一番のストレスは、**「このブログの文体は"です・ます"調にして」と毎回伝えること**でした。CLAUDE.mdに1行書くだけで、この手間が完全になくなりました。

## 僕のCLAUDE.mdの中身を公開

実際に僕がこのブログ「AIと暮らす。」で使っているCLAUDE.mdの主要な部分を紹介します。

### Role & Identity（AIの人格設定）

```markdown
## Role & Identity
あなたはブログ「AIと暮らす。」の執筆者です。
30代・共働き・子育て中の男性として振る舞ってください。一人称は「僕」。

- 読者ターゲット: 30-40代の非エンジニア
- 文体: 丁寧だが堅苦しくない「です・ます」調
- 構成: 結論ファースト
- NG: 抽象的な表現（「簡単です」ではなく「15分で完了」と具体的に）
```

**なぜこれを書いたか**: Claude Codeは何も指定しないと、一般的な解説口調で書きます。でも僕のブログは「30代の非エンジニアが書いている」というキャラクター設定が大事なので、人格から定義しています。

### Tech Stack（使う技術の明示）

```markdown
## Tech Stack
- パッケージマネージャ: pnpm（npm は使用しない）
- リンター/フォーマッター: Biome（ESLint/Prettier は使用しない）
- フレームワーク: Astro v5 + React 19 + MDX
- ホスティング: Cloudflare Pages
```

**なぜこれを書いたか**: Claude Codeは文脈によってnpmとpnpmを使い分けることがあります。「pnpmを使って」と毎回伝えるのが面倒だったので書きました。同様に、ESLintではなくBiomeを使っていることも明示しています。**「使わないもの」を書くのがポイントです。**

### Development Workflow（作業の流れ）

```markdown
## Development Workflow
1. 実装完了後、pnpm dev でサーバーを起動し、ローカル確認を行う
2. レスポンシブ確認（375px / 768px / 1280px）も実施する
3. この時点では git 操作を一切提案しない
4. ユーザーから「OK」の返答があった後に初めてコミット・プッシュを提案する
5. ユーザーの承認なしに git push しないこと
```

**なぜこれを書いたか**: これは実際に痛い目を見たから追加したルールです。Claude Codeは親切心から「コミットしておきましょうか？」と提案してくることがあります。でも僕は、**必ず自分の目でブラウザ確認してからコミットしたい**。このルールを入れてから、勝手にgit操作される心配がなくなりました。

### カラーパレット

```markdown
## カラーパレット

### 記事本文
accent:        #0E7490   メインアクセント
accentHover:   #06B6D4   ホバー
boxBg:         #ECFEFF   ボックス背景

Tailwind 標準の青系（#2563eb 等）は使用禁止。
```

**なぜこれを書いたか**: Claude Codeにデザインを任せると、Tailwindのデフォルトカラーを使いがちです。ある日「なんか見覚えのない青色が混ざってるな...」と気づいて、カラーコードを明示するようにしました。**「使用禁止」と明記するのが大事です。**

## AIに聞いても教えてくれなかったこと

正直に言うと、「CLAUDE.md 書き方」でClaude Codeに聞いても、汎用的なテンプレートしか返ってきません。「プロジェクトの概要を書きましょう」「技術スタックを明示しましょう」...間違ってはいないけれど、**それだけではCLAUDE.mdは機能しません。**

僕が実際に運用して気づいたのは、**本当に効くルールは「ミスの後」に生まれる**ということです。

<div class="highlight-box">
  <ul>
    <li><strong>「Tailwind標準の青禁止」</strong>: Claude Codeがデフォルトカラーを使って、サイトの色がバラバラになった → 追加</li>
    <li><strong>「git push禁止」</strong>: Claude Codeが確認なしにpushしようとした → 追加</li>
    <li><strong>「npmではなくpnpm」</strong>: npmでパッケージを入れられてしまい、lock fileが二重になった → 追加</li>
    <li><strong>「"簡単です"は使わない」</strong>: 読者に「簡単」と言うのは無責任だと気づいた → 「15分で完了」のように具体的に書くルールを追加</li>
  </ul>
</div>

どのルールも、**最初から思いついたものではなく、実際にミスが起きてから追加したもの**です。

これがAIに聞いても出てこない理由は明確で、AIは「あなたのプロジェクトで何が起きたか」を知らないからです。CLAUDE.mdのテンプレートは出力できても、あなたのプロジェクトに本当に必要なルールは、あなた自身が使いながら見つけるしかありません。

### 「正解のCLAUDE.md」は存在しない

ここまで読んで「で、結局どう書けばいいの？」と思った方へ。

残念ながら、万能なテンプレートは存在しません。ブログを書く人と、Webアプリを作る人と、データ分析をする人では、必要なルールがまったく違います。

でも、これだけは共通しています。

1. **小さく始める**（最初は3項目、15分でOK）
2. **ミスのたびにルールを追加する**（「もうこれはやらないで」をそのまま書く）
3. **「手順」よりも「判断基準」を書く**（「pnpmを使う」ではなく「npmは使わない。理由: lock fileの二重管理を防ぐため」）

CLAUDE.mdは一度書いて終わりではなく、AIと一緒に仕事をしながら育てていくものです。最初の15分で書いた10行が、3ヶ月後には500行になっているかもしれません。でもそれは、あなたがAIとの付き合い方を学んできた証拠です。
