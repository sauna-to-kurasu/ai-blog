## 結論：失敗はAIが直してくれた。でも「なぜハマったか」は自分にしかわからない

最初に言い切ります。

プログラミング経験ゼロの僕がClaude Codeでブログを2サイト作りました。その過程で、数えきれないほどの失敗をしています。開発ログは85エントリ以上。**その大半が「ハマった→解決した」の記録です。**

嬉しいことに、エラーはほぼ全部AIが解決してくれました。でも、**「なぜそのエラーが起きたのか」「次から何に気をつければいいのか」は、AIには教えてもらえませんでした。**

## 失敗1：「全部お任せ」で指示したらカオスが返ってきた

### 何が起きたか

ブログを作り始めた初期、僕はClaude Codeにこんな指示を出していました。

<AiChat messages={[
  { role: "human", text: "いい感じのブログサイトを作って" },
  { role: "ai", text: "承知しました！ブログサイトを作成します。まず、デザインフレームワークとして...(大量のコードを生成し始める)" }
]} />

結果、大量のファイルが生成されて、何がどうなっているのかまったくわからない状態に。色も構成もバラバラ。しかも一部のファイルはビルドエラーで動かない。

### なぜハマったか

**「全部お任せ」は、AIにとって「何をしていいかわからない」と同じ意味でした。** 人間同士の会話では「いい感じに」で通じることもありますが、AIは具体的な指示がないと「考えられる全パターン」を詰め込もうとします。

### 教訓

<div class="highlight-box">

**具体的に伝えるだけで、手戻りは激減する。** 「ヘッダーをダークブルーの背景で、ロゴを左寄せ、ナビを右寄せにして」のように、1回の指示を1つのタスクに絞る。これだけで体感の手戻りが70%から10%に減りました。

</div>

## 失敗2：権限エラーでプロジェクトが作れない

### 何が起きたか

一番最初の作業、`npm create astro@latest` でプロジェクトを作ろうとしたら、いきなり権限エラー（EPERM）が出ました。

```
EPERM: operation not permitted, access '/usr/local/lib/node_modules/npm/'
```

プログラミング経験ゼロの僕には、これが何のエラーなのかすらわかりませんでした。

### なぜハマったか

Claude Codeはセキュリティのためにサンドボックス（実行制限）がかかっています。npmのグローバルモジュールにアクセスするには、その制限を解除する必要がありました。**でも「サンドボックス」という概念自体を知らないので、エラーの意味がわからなかった**のです。

### 教訓

<div class="highlight-box">

**権限系のエラーが出たら、AIにエラーメッセージをそのまま貼ればいい。** 僕の場合はClaude Codeが「権限を許可しますか？」と聞いてくれて、「はい」を選ぶだけで解決しました。エラーの意味がわからなくても大丈夫。

</div>

## 失敗3：ローカルでは完璧なのに、デプロイしたら真っ白

### 何が起きたか

自分のMacではデザインが完璧に表示されていました。意気揚々と `git push` して本番デプロイ。ブラウザでアクセスしたら、**スタイルが一切当たっていない真っ白な画面**。

### なぜハマったか

原因はTailwind CSSの依存関係が `package.json` に記載されていなかったこと。ローカルではたまたまグローバルインストールされていたので動いていましたが、デプロイ先（Cloudflare Pages）では `npm ci` でクリーンインストールするため、記載のないパッケージは入りません。

**「自分のPCで動いている＝本番でも動く」は幻想でした。**

### 教訓

<div class="highlight-box">

**新しいパッケージを入れたら、`package.json` に入っているか確認する。** AIに「依存関係がpackage.jsonに正しく記載されているか確認して」と聞くだけで防げます。この一言を覚えただけで、デプロイ失敗がゼロになりました。

</div>

## 失敗4：CSSが効かない！グローバルCSS vs Tailwindの優先度バトル

### 何が起きたか

トップページの幅を `max-w-7xl` に変更するようAIに指示しました。コードは変更された。でも**ブラウザで見ても幅が変わらない。**

何度やっても同じ。AIに「反映されない」と伝えると「ブラウザをリロードしてください」と言われる。リロードしても変わらない。

### なぜハマったか

原因は `global.css` に `main { width: 720px; }` が直接書かれていたこと。グローバルCSSのスタイルがTailwindのクラスより優先されていて、いくらTailwindで `max-w-7xl` を指定しても上書きされていました。

**CSSの「優先度（詳細度）」という概念を知らない僕にとって、これは完全に見えない敵でした。**

### 教訓

<div class="highlight-box">

**CSSが反映されないときは、「このスタイルが効かない原因を調査して。global.cssなど他のファイルとの競合も確認して」とAIに伝える。** 単に「直して」ではなく「原因を調査して」と言うのがポイント。AIは調査を指示すれば、ちゃんと競合を見つけてくれます。

</div>

## 失敗5：ファイルを変更したのにブラウザに反映されない

### 何が起きたか

画像ファイルを差し替えました。コードも正しい。開発サーバーも動いている。**でもブラウザでは古い画像のまま。**

キャッシュバスターを追加（`?v=2`、`?v=3`）しても変わらない。ハードリロード（`Cmd + Shift + R`）しても変わらない。シークレットウィンドウを開いても変わらない。

結局、開発サーバーを完全に停止→再起動してようやく反映されました。

### なぜハマったか

ブラウザのキャッシュ、開発サーバーのキャッシュ、場合によってはCDNのキャッシュ。**キャッシュの層が複数あることを知らなかった**ので、1つだけクリアして「直らない」と思い込んでいました。

### 教訓

<div class="highlight-box">

**変更が反映されないときの確認順序**: (1) ブラウザのハードリロード → (2) シークレットウィンドウで確認 → (3) 開発サーバーを再起動 → (4) デプロイ後ならCDNキャッシュも確認。この順番を覚えておくだけで、キャッシュ地獄から抜け出せます。

</div>

## 失敗6：日本語ファイル名でパスが通らない

### 何が起きたか

サウナブログを作っていたとき、画像ファイルが `25-12-30 サウナと暮らす_ファビコン.webp` という名前でした。このファイルをコードから参照しようとしたら、**パスが通らずにエラー**。

### なぜハマったか

プログラミングの世界では、ファイル名に日本語やスペースを入れるのはトラブルのもとです。でも普段PCを使う分には日本語ファイル名で何の問題もないので、**「ファイル名にルールがある」という感覚自体がなかった**のです。

### 教訓

<div class="highlight-box">

**ファイル名は英数字・ハイフン・アンダースコアだけにする。** 日本語、スペース、全角文字は使わない。最初にこのルールだけ守れば、この手のエラーは一生出ません。

</div>

## 失敗7：AIが提案したコードが自分の環境では動かない

### 何が起きたか

記事に吹き出し（会話形式のUI）を入れたくて、AIに聞きました。返ってきたのは `<!-- wp:loos/balloon -->` というWordPress形式のコード。

**僕のブログはAstro製。WordPressのコードは動きません。**

### なぜハマったか

AIは膨大な学習データから「吹き出しの実装方法」を引っ張ってきますが、「今のプロジェクトがどのフレームワークで動いているか」を常に正しく把握しているわけではありません。特に初期の頃、プロジェクトの情報をCLAUDE.mdに書いていなかったので、AIが文脈を見失いやすい状態でした。

### 教訓

<div class="highlight-box">

**AIが生成したコードは、自分の環境で使えるか必ず確認する。** そしてCLAUDE.mdに「このプロジェクトはAstro v5 + React 19で動いている」と書いておけば、AI側も正しい技術スタックで提案してくれるようになります。

</div>

## 失敗8：レスポンシブ対応のモグラ叩き。スマホ直したらPCが崩れる

### 何が起きたか

スマホ表示でテキストが画面からはみ出していたので、AIに修正を依頼。直った。でもPC表示を確認したら、今度はレイアウトが崩れている。PCを直したらタブレットがおかしい。**終わらないモグラ叩きでした。**

### なぜハマったか

「スマホでこう見えるのを直して」という指示だけでは、AIはスマホの表示だけを直します。**他のサイズへの影響までは自動で考えてくれませんでした。**

結局、`text-4xl md:text-6xl` のようなレスポンシブ指定を、375px→768px→1280pxの3つのサイズで同時に確認しながら調整する必要がありました。

### 教訓

<div class="highlight-box">

**レスポンシブの修正を頼むときは「375px、768px、1280pxの3サイズで確認して」と必ずセットで伝える。** 1サイズだけ直しても、他が崩れていたら意味がありません。

</div>

## 失敗9：gitの操作ミスで作業した内容が消えた

### 何が起きたか

ある日、デザインの大幅な変更をしていて、途中でうまくいかなくなりました。AIに「元に戻して」と伝えたところ、`git checkout .` が実行されて**コミットしていなかった変更が全部消えました。**

### なぜハマったか

gitの「コミット」と「ステージング」の概念を理解していなかった僕にとって、「元に戻す」は「直前の状態に戻す」のつもりでした。でもAIは「最後のコミット時点に戻す」と解釈しました。**コミットしていなかった数時間分の作業が一瞬で消えた**のです。

### 教訓

<div class="highlight-box">

**作業の区切りごとにコミットする。** 「完成してからコミット」ではなく「動いている状態でこまめにコミット」が正解。僕は今、30分に1回はコミットするようにしています。「AIに何か大きな変更を頼む前にコミット」も鉄則です。

</div>

## 失敗10：CLAUDE.mdを書かずに毎回同じ説明をしていた

### 何が起きたか

Claude Codeを使い始めて最初の1ヶ月、毎回のセッションで同じことを伝えていました。

- 「このプロジェクトはAstro v5で動いています」
- 「CSSはTailwind CSS v4を使っています」
- 「記事は.mdxファイルです」
- 「一人称は "僕" で書いてください」

**毎回5分かけて前提条件を説明してから、本題に入る。** 非効率だとは思いつつ、他のやり方を知りませんでした。

### なぜハマったか

CLAUDE.mdの存在を知らなかったのです。CLAUDE.mdはClaude Codeが自動で読み込むプロジェクトの説明書。ここに書いておけば、**毎回説明しなくてもAIがプロジェクトの文脈を理解した状態で会話を始めてくれます。**

### 教訓

<div class="highlight-box">

**CLAUDE.mdは初日に作る。** 技術スタック、命名規則、文体ルール、やってはいけないことを最初に書いておく。あとは使いながら「これも伝えておきたい」と思ったことを追記していけば、勝手にAIの精度が上がっていきます。

</div>

## AIに聞いても出てこなかった「失敗の本質」

10個の失敗を振り返って気づいたことがあります。

**エラーの「解決方法」はAIが教えてくれる。でも「なぜ自分がハマったのか」はAIには見えていない。**

たとえばCSS競合の失敗（失敗4）。AIに「CSSが効かない」と伝えれば原因を調べて直してくれます。でも、「グローバルCSSの優先度」という概念を知らなかった僕が**次から同じ失敗をしないために何を学ぶべきか**は、自分で気づくしかありませんでした。

10個の失敗に共通しているのは、こういうパターンです。

1. **知らない概念にぶつかる**（サンドボックス、CSSの詳細度、キャッシュの階層、gitのステージング...）
2. **AIが目の前のエラーを解決してくれる**
3. **同じ種類の失敗をもう1〜2回繰り返す**
4. **「あ、これ前もやったな」と気づいて、ようやく概念が身につく**

AIは「答え」をくれますが、「問いの立て方」は教えてくれません。**でも、失敗を2〜3回繰り返すうちに「問い」が自分の中にできてくる。** そうなれば、AIへの指示も的確になって、失敗の連鎖が止まります。

だからこそ言いたいのは、**失敗を恐れなくていい**ということ。AIがいる時代、失敗のコストは劇的に下がっています。解決までの平均15分。その15分の積み重ねが、確実に「問いの力」を育ててくれます。
