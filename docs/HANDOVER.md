## ブラッシュアップの進め方

1. 記事を読んで「ここもっと聞きたい」ポイントを4つ前後質問する
2. ユーザーが口頭（音声入力）で回答する
3. 回答を記事に反映する
4. 新しい生の声は `docs/user-voice.md` に記録する（深掘り回答だけでなく、ちょっとした会話・指摘・補足の中の生の声も取りこぼさない）
5. `pnpm dev` で確認 → ユーザーOK → ブランチ→PR→マージ

## 今回のセッションで得た教訓

- **AIが推測で書いた心情は要注意**: No.4の「60%で出す」セクションがAIの推測で、実態と乖離していた。深掘り質問で発見できたケース。初稿のAI推測心情は必ず本人に確認すること
- **記事の大前提レベルのずれもある**: No.5は「人間がCLAUDE.mdを書く」前提で全体が構成されていたが、実際はAIに全部任せていた。深掘り質問4つ中3つが「書いていないので答えられない」で、根本的な方向転換が必要になった
- **会話中の生の声を取りこぼさない**: 深掘り質問への回答だけでなく、指摘・補足・つぶやきの中にも重要な生の声がある。見逃さず`docs/user-voice.md`に記録する
- **定量データはMCPで最新を取得**: GA4・GSCのデータはMCPツールで確認可能。記事内の数字を最新に更新できる
- **No.9で7セクション中5つが作り話だった**: AIが「非エンジニアあるある」系の失敗談を捏造する傾向が明確。特にプロンプトの失敗談はAIにとって「作りやすい」ジャンル
- **memoryにルールを書いてもCLAUDE.mdになければ実行されないことがある**: 構造的な問題。次セッションでルールの最適配置を検証する
